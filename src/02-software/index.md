# 软件系统

上一章建立了一个关键判断：供给侧约每两年 5–6 倍的系统算力增长，是每一代架构师在物理约束下通过引入新设计维度、将帕累托前沿外推的累积结果。但硬件前沿外推只打开了可能性空间；**把新的硬件可能性稳定兑现为 Goodput，是软件系统的任务。** 同样的互联带宽、同样的显存容量，在不同的软件栈与运行时策略下，可能呈现出完全不同的吞吐、尾延迟与可运维性。前沿在硬件层面被推出多远，与前沿在真实业务中被兑现多少，中间的差距完全由软件决定。从这个意义上说，**超节点首先是一个软件问题**。

本章沿着这条线索展开。前三节以"**统一内存子系统**"为整体，沿**定义能力 → 落地能力 → 调度能力**三层递进：首先建立 Scale-Up 域内内存语义、保序与地址翻译的通用分析框架，回答"硬件必须提供哪些语义能力"；然后以 NVIDIA NVLink/NVSwitch Fabric 为例，展示这些通用能力如何通过 GMMU 页表、Fabric 地址空间和 Export/Import 接口落地为具体实现；内存池化将视角拉升到系统软件层，讨论如何在多层级异构存储间高效调度，把统一寻址能力转化为 Goodput 提升。后续三节将视野扩展到更广的软件栈：通信运行时与编程模型把带宽转化为稳定吞吐；可观测性与 RAS 体系应对大规模系统中无处不在的灰色故障和性能退化；训练与推理系统工程则把上述能力落到并行策略、缓存分层、调度与运维流程中。Goodput 不是单点优化的结果——它需要把硬件语义、寻址、通信、RAS 与训推工程作为一条连续链路来设计。而规模化系统的真正难点恰恰在于稳定兑现：拥塞、抖动、灰色故障与跨域收敛问题，往往比峰值带宽本身更影响实际效率。

但软件系统中的很多取舍并不能靠经验直接完成，需要通过建模、仿真和校准来量化"当前距帕累托前沿还有多远、哪个维度是最紧约束"。因此下一章将进入建模仿真，讨论如何把架构争论转化为可验证假设。
