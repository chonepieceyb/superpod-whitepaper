# SuperPod Pareto Index

`SuperPod Pareto Index (SPI)` 是面向超节点产品的开放式提交与委员会评估框架。它不是一个类似 TOP500 的单榜单，而是一套用于描述超节点产品在统一坐标系中相对位置的方法。这里的坐标系，本质上是在描述产品位于当前**系统能力边界**的什么位置；在方法论上，这条边界继续使用帕累托前沿来形式化。

SPI 的目标有两个：

1. 建立一个由厂商通过 GitHub Pull Request 提交的标准化产品名录。
2. 基于白皮书现有七维框架，由编写委员会对产品形成统一评分、雷达图与简短评述。

## SPI 的两层结构

- **SPI Registry**：记录厂商提交的结构化 `spec`、证据支持的客观数字和厂商自述。
- **SPI Scorecard**：由编写委员会依据七维框架给出评分和雷达图，形成可横向比较的多维定位结果。

这意味着 SPI 不追求“谁是绝对第一”，而是强调：**谁在什么约束下接近当前系统能力边界，谁又在什么维度上做出了让步。**

## 为什么不是 TOP500

TOP500 建立在单一指标（Linpack）的统一排序上，而超节点竞争本质上是多维权衡问题。只比较理论算力、峰值带宽或卡数，都会掩盖真正影响 Goodput 的关键因素。SPI 的目的，就是把这些 trade-off 放回同一个决策面上，而不是压缩成一个总分。

SPI 因此沿用本白皮书现有七维框架：

1. 单跳/访存延迟
2. 规模上限
3. 拓扑弹性
4. 生态成熟度
5. 功耗与 TCO
6. 软件复杂度
7. 适用负载

其中前六项原则上采用 1-5 分制，`适用负载` 以标签与评语为主。

## 厂商如何参与

SPI 的默认参与方式是 **GitHub Pull Request**。

厂商提交时可以提供三类信息：

- **客观规格**：产品结构、互联方式、规模、功耗、软件栈等
- **证据支持的客观数字**：吞吐、延迟、带宽、Goodput、功耗等，但必须说明测试条件并提供证据链接
- **厂商说明**：解释该产品的优势、目标场景和技术亮点

需要强调的是：

- 厂商可以提交数字，也可以解释数字；
- 但最终的评分、雷达图与委员会评语，不由厂商直接决定。

## 委员会职责

编写委员会在 SPI 中承担三类职责：

- 审核厂商提交内容的完整性与证据等级
- 基于统一七维框架给出评分
- 形成可公开追溯的委员会评语与版本记录

这样做的目的，是让厂商展示自己的“肌肉”，同时避免 SPI 退化为单纯的营销参数表。

## 仓库结构

当前 SPI 已在仓库中预留基础结构：

- `spi/README.md`
- `spi/templates/product-template.yaml`
- `spi/schema/spi-product.schema.json`
- `spi/products/`

厂商提交和委员会评审将围绕这一结构逐步展开。

## 已收录产品入口

当前已收录产品不再混在 SPI 总览中，而是通过独立导航页展示：

- [当前已收录产品](07-spi-registry.md)

该页面在构建时自动扫描 `spi/products/` 目录生成，因此会随着仓库中新增产品条目而更新。

## 后续演进

SPI 可以逐步扩展为：

- 白皮书附录中的正式方法论章节
- 站点中的产品名录与雷达图页面
- 面向产业界的开放提交与周期性更新机制

如果做成，它将不只是白皮书的附属功能，而会成为超节点领域的一个公共坐标系。
