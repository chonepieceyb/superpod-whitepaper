# 未来演进

第四章给出了当前系统能力边界上的五个参考设计。但这些方案的成立条件并非静态——它们依赖的互联形态、封装路径、内存架构和负载画像都在变化。本章不再罗列未来技术，而是回答一个更直接的路线判断问题：**在未来 2–3 年内，哪些技术变量最可能以可工程化的方式改变超节点系统能力边界的主导约束，并因此重写当前参考设计的优先级？**

为什么是 2–3 年？因为 NVIDIA 的代际路线图提供了一条清晰的淘汰基准线：从 Hopper 到 Blackwell 到 Rubin，系统级推理性能每代提升约 10×（硬件 ~2× × 软件 ~5×），训练效率的 GPU 数量需求每代缩减至约 1/4。任何超节点架构选择，如果不能在这一窗口期内把系统能力边界向外推一个数量级，就会被下一代平台直接替代。因此，本章对每一项技术变量的讨论，都要落回两个判断：**它先改写什么约束？它会如何改变第四章的方案优先级？**

## 哪些变量会改写超节点能力边界

能力边界不只是"向外推"——它的坐标轴本身也在变化。当工作负载从稠密训练演进到推理-reasoning 混合、长上下文 MoE、多模态 Agent 时，"tokens/s/user vs tokens/s/GPU"不再是唯一重要的权衡面。新的关键维度正在浮现：能效（tokens/s/MW）、上下文容量（KV cache $/token）、可重构性（动态拓扑使能力边界从静态曲线变为动态曲面）。在这些新维度浮现的同时，三组技术变量正在改写既有约束：

**互联与拓扑**：当 SerDes 速率提升、链路预算趋紧时，铜互联在插损、功耗、可维护性上的压力会迅速放大。光互联与光交换的引入，本质上是在更大物理尺度上把"带宽与抖动"重新做可控化，同时为可重构拓扑提供物理抓手。LPO → NPO → CPO 的演进不是简单的"一代比一代好"，而是在功耗、集成度、可维护性与成熟度之间反复取舍。它先改写的约束是**链路功耗与拓扑灵活性**。

**封装与近存储**：算力密度的提升越来越依赖 2.5D/3D 集成与堆叠内存（HBM3e → HBM4 → 3D DRAM）。它们把互联距离压缩到封装内，却把供电、散热、良率与测试的复杂度推到台前。Chiplet + UCIe 的模块化路径在算力密度、良率、成本与互联带宽之间寻找工程可行解。它先改写的约束是**算力密度与内存容量/成本**。

**模型与负载形态**：低精度（NVFP4）、长上下文、稀疏/MoE、多模态，会把系统压力从"峰值算力"转移到"内存带宽/容量、通信小包与尾延迟、以及调度与资源隔离"。模型技术的演进不是独立话题，而是用来定义互联、封装与系统软件的约束条件——它决定了当前能力边界上哪些区域是当下最需要优化的。它先改写的约束是**瓶颈位置本身**。

## 这些变量会先改变什么约束

上表的一个关键读法是**时间排序**，而不是技术分类。

**当前已在改变的约束**：低精度和模型形态变化已经在重排系统瓶颈的优先级——从"谁的峰值算力高"转向"谁的内存带宽、尾延迟和调度效率更优"。LPO 光模块已进入量产，正在改善全部方案的链路功耗。软件栈的快速迭代已经证明，同一代硬件上的 Goodput 可以在数周内提升数倍。这些不是"未来趋势"，而是"已经在影响今天方案选择的变量"。

**1–2 年内将改变的约束**：HBM4 和 3D DRAM 将显著缓解内存容量/成本瓶颈，使得更大规模的长上下文推理和 KV Cache 池化方案变得经济可行。OCS 光交换的原型验证正在推进，如果控制面成熟度跟上，它将使 Dragonfly+OCS、Torus+OCS 等探索构型从"概念可行"走向"工程可部署"。这一时间窗是探索构型从"并行验证"转为"规模试点"的关键判断点。

**2–3 年后可能改变的约束**：CPO（共封装光学）和 Chiplet + UCIe 有望在功耗密度和算力密度上带来结构性突破，但两者都面临散热、良率和供应链的工程化挑战。可重构拓扑的控制面框架目前仍处于早期，距离生产级部署还有明显差距。这些变量值得持续跟踪和投入，但不应作为当前部署的前提假设。

**证据等级的意义**：第三章建立的仿真方法论，在这里有直接的应用价值——对于"已验证趋势"，仿真模型可以直接校准；对于"工程推断"，仿真可以量化其对边界位置的预期改变幅度；对于"方向判断"，仿真的作用更多是设定监测指标和拐点阈值，而非给出确定性结论。

## 对当前参考设计与产业路线意味着什么

回到第四章的五个参考设计，上述变量会如何改变它们的优先级？

### 被强化的前提

- **标准以太网方案**：LPO/NPO 的成熟将持续改善其链路功耗，开放生态与以太网 PHY 兼容性使其天然受益于光模块代际演进。MoE/长上下文负载对 EP/DP 带宽的依赖增强，而非对单跳延迟的极致要求，进一步巩固了以太网方案在这类场景下的竞争力。
- **标准总线方案**：Chiplet + UCIe 的推进将直接强化总线语义的价值——如果 UCIe 物理层与 UALink 协议层融合成功，总线方案将获得一条从 Scale-Up 向 Chiplet 级延伸的完整内存语义链路。HBM4 对内存容量的提升也有利于总线方案擅长的细粒度显存共享场景。

### 被削弱的前提

- **所有方案**都面临一个共同风险：如果模型形态变化的速度快于预期（例如超稀疏 MoE 或多模态 Agent 成为主流负载），当前参考设计中基于稠密训练假设的带宽分配和拓扑设计可能需要根本性调整。
- **探索构型**中依赖 OCS 的方案（Dragonfly+OCS、Torus+OCS、Ring+dOCS），其成立的关键前提是光交换控制面的成熟度。如果控制面在 1–2 年内未能达到生产级要求，这些方案将停留在验证阶段而无法转入规模部署。

### 部署策略再判断

结合上述分析，第四章"成熟路径优先、探索路径并行验证"的总策略仍然成立，但可以更精细地分层：

| 部署层级 | 参考设计 | 依据 | 关键验证指标 |
|:---------|:---------|:-----|:-----------|
| **近期主力** | 标准以太网、标准总线 | 生态成熟、供应链就绪、LPO/HBM4 直接受益 | Goodput 实测、软件栈覆盖度 |
| **并行验证** | Dragonfly+OCS、Torus+OCS | OCS 原型可用、拓扑弹性价值已在仿真中验证 | OCS 控制面延迟、故障恢复时间 |
| **中期储备** | Ring+dOCS | dOCS 去交换化概念领先，但控制面与器件成熟度最远 | dOCS 器件良率、端到端延迟 |

### 编写委员会建议持续跟踪的判断指标

- **OCS 切换延迟**是否降至 ms 级以下，决定探索构型能否从验证转入部署。
- **CPO 功耗密度**是否在 2–3 年内达到 LPO 的 50% 以下，决定超高密部署方案的可行性。
- **主流训练负载中 MoE/长上下文占比**是否超过 50%，决定带宽分配假设是否需要根本调整。
- **软件栈对新硬件能力的承接速度**（以 Goodput 提升倍数 / 硬件上市后月数衡量），决定"硬件打开上限、软件决定落地"这一判断的持续有效性。
