# 编写委员会

## 组织原则

超节点技术横跨芯片、互联、封装、软件和系统集成等多个产业环节，涉及不同厂商的技术路线与商业利益。为确保白皮书的**专业性、中立性与权威性**，本白皮书采用"学术主导编辑、产业分域贡献"的组织机制：

- **编辑权归属学术方**。编写委员会由来自高校和科研机构的研究者构成，负责全书的框架设计、稿件整合、技术审校与质量控制。编委会对最终发布内容拥有独立的编辑判断权，不受任何单一产业方的技术路线或商业立场约束。
- **产业方按技术方向贡献内容**。产业伙伴在各自专长的技术领域提供素材、数据、案例和讨论意见，其贡献经编委会审校后纳入正文。这一机制使白皮书能够充分吸收一线工程经验，同时避免任何单一厂商的技术偏好主导全书判断。
- **顾问委员会提供战略指导**。由跨领域的资深专家组成顾问委员会，在技术方向、分析框架和重大判断上为编委会提供独立建议，进一步保障白皮书的学术视野和判断质量。

这一分权结构的设计目标是：让白皮书成为**基于学术共识的行业公共知识产品**，而非任何单一机构的技术主张。

## 全书结构与协作框架

全书分为三个部分，分别回答三个递进问题。这一结构同时也是协作分工的框架——每个部分对贡献者的交付要求不同：

### 第一至三章（架构分析、软件系统、建模仿真）

**核心问题**：超节点为何成立，核心约束是什么。

| 要素 | 说明 |
|:-----|:-----|
| **交付形态** | 技术分析、机制解释、定量论证 |
| **核心要求** | 每一节必须服务于帕累托前沿外推的主线——要么证明某个约束的存在，要么解释某个维度上前沿被推移的机制 |
| **不做什么** | 不做技术百科或产品手册；不需要覆盖所有细节，但必须在因果链上闭环 |
| **输出供** | 第四章参考设计（前三章的分析直接构成参考设计的评估依据） |

### 第四至六章（参考设计、未来演进、总结）

**核心问题**：当前如何选择工程路径，未来哪些变量将改写前沿。

| 要素 | 说明 |
|:-----|:-----|
| **交付形态** | 方案对比、帕累托定位、路线判断、行动建议 |
| **核心要求** | 参考设计必须给出可比较的帕累托定位（七维度评分表）；未来演进必须回答"对参考设计意味着什么"；总结必须面向不同读者给出可执行的下一步 |
| **不做什么** | 不做开放式讨论；每个判断必须有前三章的分析依据支撑 |
| **输出供** | SPI 筹备（本部分的七维分析框架为后续产品条目、证据口径与治理讨论提供方法论基础） |

### SPI 筹备与产业生态地图

**核心问题**：如何把分析框架进一步转化为可共建的协作接口。

| 要素 | 说明 |
|:-----|:-----|
| **交付形态** | 筹备说明、问题清单、预留提交流程、产业协同地图 |
| **核心要求** | SPI 要诚实说明当前还没定下来的治理、证据、边界与版本问题；生态地图必须识别关键环节、短板位置和协同接口 |
| **不做什么** | 不把 SPI 写成已经定稿的正式机制；不把样例条目包装成成熟评估结果 |
| **输出供** | 外部产业协作（本部分是白皮书从分析框架走向产业共建的接口） |

### 跨章协作原则

- **每章结尾必须回答"对参考设计意味着什么"**。这是全书收束的关键机制——无论贡献者撰写的是前三章的技术分析还是未来演进，最终都要汇入第四章的决策框架。
- **证据分级标记**。所有技术判断应标注证据等级：`已验证`（有公开实测数据支撑）、`工程推断`（基于架构分析或代码逻辑）、`方向判断`（基于趋势外推）。这使读者能区分确定性不同的结论，也使筹备中的协作议题可以诚实标注当前成熟度。
- **章节合同制**。每章在正式撰写前，由编写委员会确认五项内容：本章只回答什么问题、关键结论是什么、必须交付什么形态、输出供哪一章使用、不讨论什么。章节合同经确认后作为内容审校的基准。

## 数据与素材贡献

本白皮书的分析建立在公开文献、技术规范与参与单位提供的讨论素材基础上。以下类型的数据和分析对白皮书形成了关键支撑：

| 数据类型 | 价值说明 |
|:---------|:---------|
| 超节点架构细节与代际演进数据 | 支撑帕累托前沿外推机制的定量分析 |
| 光模块功耗与性能实测数据 | 支撑 LPO/CPO 技术路径的工程判断 |
| 先进封装工程数据 | 支撑 2.5D/3D 封装路径的可行性评估 |
| 大模型训练负载通信画像 | 支撑 MoE/长上下文等新范式下的需求建模 |
| 集群运维与故障数据 | 支撑 RAS 体系与 Goodput 兑现的分析 |
