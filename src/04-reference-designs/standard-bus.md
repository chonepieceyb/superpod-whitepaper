# 标准构型：总线全对等互联方案

以 UALink（Ultra Accelerator Link）为协议基础，通过专用交换芯片在机柜内构建扁平内存地址空间，使加速器间可直接执行 Load/Store/原子操作，形成一个逻辑上接近"放大版单机"的高带宽域。该方案的核心取舍是：**以原生内存语义和最低访存延迟为优先约束，在此约束下尽可能维持开放生态的供应链灵活性**。

## 目标问题

以太型 Scale-Up 方案（ESUN/SUE）通过在以太网协议栈上叠加内存语义来逼近总线性能，但其本质仍是"把网络改造成总线"。对于 TP（张量并行）这类对访存延迟极度敏感的通信模式，即使 ESUN 的 250–500 ns 单跳延迟仍显不足——NVLink 以 <100 ns 的单跳延迟维持着显著优势，原因在于它从一开始就是按内存总线而非网络协议设计的。

UALink 正是要在开放标准框架下提供这种"原生总线"能力。它不复用以太帧格式，而是采用专用 Flit 设计，从协议栈底层就为内存语义访问优化。对于希望获得接近 NVLink 性能、又不愿被锁定在单一供应商生态中的加速器设计者而言，UALink 是当前最有竞争力的开放选项。

## 协议基础：UALink 1.0

UALink 由 AMD 牵头，联合 Intel、Google、Meta、Microsoft、HPE 等于 2024 年发起，2025 年 6 月发布 1.0 规范。其设计哲学可以概括为：**用总线的方式做互联，而非用网络的方式模拟总线**。

核心技术特征：

- **FAM（Flat Address Memory）架构**：所有加速器的显存被映射到一个全局扁平地址空间，任意加速器可对任意远端显存执行 Load/Store/原子操作，无需显式消息传递
- **固定 640B Flit 设计**：不使用以太帧格式，而是采用固定长度的微包（Flit），消除帧头解析开销，使转发路径极度精简
- **单通道 200 GT/s，四通道 800 Gbps 全双工**：物理层采用以太网 SerDes（复用成熟 PHY IP），但链路层完全自定义
- **64B 负载往返延迟 <1 μs**：包含协议处理、交换转发与链路传输的端到端延迟
- **链路利用率 93%**：得益于 Flit 设计与 LLR + CBFC 机制，有效载荷率远超标准以太帧
- **单 Pod 规模 ≤1024 加速器**：1.0 规范定义单级交换域

与 ESUN 的关键区别在于：ESUN 在以太网帧格式上增强，保持与以太交换机的兼容；UALink 则彻底放弃以太帧，用专用 Flit 换取更低的转发延迟和更高的有效载荷率。两者在物理层共享以太网 SerDes，但链路层与事务层的设计哲学完全不同。

## 架构定义与组网方式

该构型采用"加速器 + UALink 交换芯片"的两级结构：

- **计算节点层**：GPU/XPU 通过片上或近端 UALink PHY 直接连接 UALink 交换芯片。每个加速器暴露一段 FAM 地址空间，交换芯片负责地址路由与事务仲裁。
- **UALink 交换层**：专用 UALink 交换芯片（类比 NVLink 体系中的 NVSwitch）提供 Flit 级全交叉互联，将 Pod 内所有加速器的 FAM 地址空间统一编址。

在机柜级部署中，典型配置为：

| 参数 | 典型值 |
|:-----|:------|
| 单 Pod 加速器规模 | 64–1024 XPU（UALink 1.0） |
| 单链路带宽 | 800 Gbps 全双工（四通道） |
| 交换层级 | 1 级（单级全交叉） |
| 内存语义 | 原生 Load/Store/Atomic（FAM） |
| 64B 负载 RTT | <1 μs |
| 链路利用率 | 93% |
| 地址空间 | 全局扁平（FAM），硬件一致性可选 |

UALink 1.0 定义的单级交换域在规模上可覆盖当前主流超节点需求（64–1024 卡）。当需要进一步扩展到多 Pod 时，可通过 Scale-Out 网络（InfiniBand / RoCEv2 / UEC）连接多个 UALink Pod，形成"Pod 内总线 + Pod 间网络"的两级架构。

## 核心组件

### UALink 交换芯片

UALink 交换芯片是该构型的核心瓶颈与关键依赖。其功能类比 NVSwitch，但基于开放规范：

- 全交叉 Flit 交换，支持 FAM 地址路由
- LLR + CBFC 硬件卸载
- 多播 / 规约操作硬件加速（对 AllReduce 等集合操作至关重要）

**当前状态**：截至 2025 年，AMD 自身尚未公布量产的 UALink 交换芯片产品。这是该方案落地的最大不确定性——协议规范已就绪，但硬件供应链尚未完全成熟。Broadcom、Marvell 等交换芯片厂商是潜在的供应来源。

### 加速器侧 UALink PHY

加速器需要在片上或封装内集成 UALink 物理层接口。由于 UALink PHY 复用以太网 SerDes IP，硅片设计的额外成本可控。AMD MI 系列后续产品预计将原生集成 UALink 端口。对于国产加速器而言，UALink PHY 的集成难度低于自研全栈专有互联。

### 互联介质

Pod 内互联距离通常 <3m，铜缆 DAC 即可满足需求。UALink 物理层与以太网 SerDes 兼容，因此可复用标准 400G/800G 铜缆与连接器，降低物料成本。

## 软件与控制面

总线型方案的软件栈与以太型存在根本差异——编程模型更接近共享内存，而非消息传递：

- **统一地址映射**：运行时需要管理 FAM 全局地址空间的分配与映射，使上层框架（PyTorch / JAX）可以透明地访问远端显存
- **一致性协议**：UALink 1.0 提供硬件一致性选项，但在大规模 Pod 中，完全硬件一致性的开销可能过高。实际部署需要在硬件一致性与软件管理一致性之间做权衡
- **集合通信库**：AllReduce、All-to-All 等操作可以利用 UALink 交换芯片的硬件规约能力，性能潜力优于纯软件实现。但通信库需要深度适配 UALink 的 Flit 语义与 FAM 接口
- **故障隔离与 RAS**：总线型互联中，单个加速器或链路故障可能影响整个 FAM 地址空间的一致性。RAS 设计需要在故障隔离粒度与性能之间寻找平衡

### 与 CXL 的关系

UALink 与 CXL（Compute Express Link）在内存语义层面有显著重叠，但定位不同：

- **CXL** 面向 CPU-GPU、CPU-内存扩展等异构场景，强调与 PCIe 的兼容性
- **UALink** 专为加速器间互联设计，不需要兼容 PCIe 事务层，因此在延迟和带宽效率上有更大优化空间

在实际超节点架构中，两者可能共存：UALink 用于加速器 Pod 内互联，CXL 用于内存池化与 CPU-GPU 协同。

## 行业实践与生态

UALink 的联盟阵容覆盖了除 NVIDIA 之外的主要 AI 加速器与系统厂商：

- **AMD**：UALink 的核心推动者，MI 系列后续产品预计原生支持。AMD 此前的 Infinity Fabric 仅支持 8–16 GPU 互联，UALink 将把开放互联规模提升两个数量级
- **Intel**：Gaudi 系列加速器的后续产品有望集成 UALink，与 CXL 形成互补
- **Google**：TPU 团队参与 UALink 制定，未来可能将 TPU 的私有互联逐步向 UALink 标准靠拢
- **国内生态**：阿里 ALS（ALink System）的数据面已采用 UALink 协议（ALS-D），形成了首个国内落地的 UALink 兼容实现。这为国产加速器接入 UALink 生态提供了可参考的路径

## 帕累托视角分析

总线型方案在帕累托空间中的位置可以概括为：**在访存延迟和内存语义完整性维度上占据前沿，在生态成熟度和供应链就绪度维度上存在明显短板**。

| 维度 | 帕累托位置 | 与以太型（ESUN）的对比 |
|:-----|:----------|:---------------------|
| 访存延迟 | <1 μs RTT（64B） | ESUN 250–500 ns 单跳，但 RTT 更长 |
| 内存语义 | 原生 FAM，硬件级 Load/Store | ESUN 通过事务层映射，额外开销 |
| 有效载荷率 | 93%（Flit 设计） | ESUN 80–85%（以太帧开销更大） |
| 生态成熟度 | 交换芯片未量产，通信库早期 | 交换芯片已量产，以太生态成熟 |
| 供应链灵活性 | 交换芯片来源待定 | 多厂商以太交换机可选 |
| 运维复杂度 | 需要新的管理工具链 | 复用以太网运维体系 |
| 适用负载 | TP-heavy，细粒度显存共享 | EP/DP，大块数据传输 |

关键权衡：总线型方案为了获得原生内存语义和更低延迟，放弃了以太帧兼容性。这意味着它无法复用现有以太交换机和管理工具，需要专用交换芯片与配套软件栈。在交换芯片量产之前，该方案更多是一个"帕累托前沿上的规划位置"而非"当前可落地的工程选择"。

因此，该方案在帕累托前沿上的最佳适配区域是：**以 TP 为主的稠密训练、需要跨卡显存共享的长上下文推理、以及要求非 NVIDIA 开放生态的高性能互联场景**。

## 局限与演进方向

- **交换芯片空白**：UALink 1.0 规范已发布，但量产交换芯片是最大瓶颈。在交换芯片就绪之前，该方案只能作为前瞻规划而非即时部署选项。
- **规模扩展**：UALink 1.0 定义单级 Pod（≤1024），跨 Pod 互联需要依赖 Scale-Out 网络。后续版本是否引入多级交换或分层寻址，将决定该方案的规模上限。
- **一致性开销**：在 512–1024 卡规模下，硬件一致性协议的广播风暴可能成为延迟瓶颈。需要在目录一致性、区域一致性与软件管理一致性之间找到平衡。
- **与 ESUN 的融合可能**：UALink 和 ESUN 共享以太网 PHY，长期看两者可能在物理层融合——同一 SerDes 端口根据对端类型自适应切换 UALink Flit 模式或 ESUN 以太帧模式。这种"双模"方案将在帕累托空间中覆盖更宽的区域。
- **国产路径**：对国产加速器而言，UALink PHY 集成门槛低（复用以太网 SerDes IP），真正的挑战在交换芯片。阿里 ALS 已验证了 UALink 协议兼容性；如能联合推动 UALink 交换芯片的国产化，将为国内超节点提供一条不依赖 NVLink 的高性能互联路径。
