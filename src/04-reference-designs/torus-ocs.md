# 探索构型（3D Torus + OCS 型）

以太互联 + 3D Torus + OCS 弹性架构，面向的是大模型训练、推理与智算中心规模化场景下的一个核心问题：单卡算力持续提升，但超节点规模、通信效率和系统可用性无法同步扩展。该构型尝试通过计算、互联、拓扑与调度四层协同，把标准以太网端侧、透明光交换和可重构 Torus 拓扑结合起来，构建高带宽、低时延、高可用、易扩展的超节点底座。该方案在帕累托前沿上的核心取舍是：**以全拓扑弹性和故障隔离能力为优先约束，为此承担最高的软件协同复杂度**。

## 目标问题

自 Transformer 模型诞生以来，大模型参数规模从千亿级迈向万亿级，序列长度也从 4K 持续推向 1M。按照公开数据拟合，AI 模型训练计算量长期保持快速指数增长，而单卡算力增长则越来越难独立支撑这一趋势。因此，超节点设计的关键不再只是"把更多卡接起来"，而是如何在更大规模下继续维持有效算力、控制通信开销并减少故障对任务切片的影响。

3D Torus + OCS 的切入点在于：把标准以太网协议栈保留在节点侧，而在互联层尽量减少大型电交换机带来的层级、功耗和转发开销，通过 OCS 的动态光路切换，把物理网络重构成适配任务需求的逻辑 3D Torus。

## 架构定义与组网方式

该构型本质上是一种大平层网络设计。计算节点保留标准的以太网协议栈，例如 `RoCEv2`，输出高速光信号；互联层不再依赖传统大型电交换机逐级转发，而是让计算节点的光纤直接接入 OCS 的输入/输出端口；再由 OCS 在光层内部动态重组链路，把节点逻辑组织成首尾相连的 3D Torus 网格。

这种结构有三个直接结果：

- **计算接入层标准化**：节点侧仍可复用成熟以太网生态，降低协议和软件栈迁移成本。
- **光层全互联化**：计算节点接入 OCS 时不必经过中间电交换转发，有利于降低多跳排队与光电转换开销。
- **拓扑可重构**：当任务规模变化、切片需求调整或节点故障发生时，OCS 可以实时重构光路，改变 Torus 的形状或绕过故障节点。

从算力演进角度看，这种架构的一个重要优势是其扩展性主要受 OCS 端口规模驱动，而不是受制于高端电交换芯片的工艺与代际节奏。只要 OCS 端口数能够继续提升，超节点组网规模就有继续向上扩展的路径。

## 核心组件

### OCS 光路交换机

OCS 是该构型的物理层核心枢纽，对速率和协议保持透明。常见器件路线包括：

- **MEMS**：通过微镜偏转实现光路切换，当前商业化最成熟。
- **液晶**：无活动部件，通过改变折射率实现切换，可靠性较高。
- **压电陶瓷**：利用压电效应驱动物理对准，低插损。
- **硅光子开关**：具备纳秒级切换潜力，是未来高速光交换的重要方向。

### 光模块与光电集成

当前可采用 400G/800G 等可插拔光模块作为主流实现。随着单芯片 I/O 带宽向 T 级迈进，传统可插拔模块的功耗会越来越突出，因此中长期会逐步向 `NPO` 与 `CPO` 演进，把光引擎进一步前移到交换芯片或加速器基板附近。

### 智能网卡与协议卸载

节点侧需要智能网卡处理 `RoCEv2` 等协议，并承担拥塞控制、遥测和端侧卸载，以保证在 Torus 拓扑中仍能维持稳定的数据平面性能。

## 软件栈与控制平面

该构型并不只是一个硬件拓扑方案，真正发挥作用还依赖软件与算法协同：

- **拓扑管理器**：集中感知物理健康度和任务规模，为不同任务切割互不干扰的 3D Torus 子网。
- **OCS 控制器**：把逻辑拓扑指令翻译成底层 OCS 器件控制信号。
- **路由机制**：以太网层通常采用维度顺序路由并结合自适应机制，在保证无环路的同时尽量平衡负载。
- **集合通信算法**：当 OCS 构建出特定尺寸的 Torus 后，通信库需要生成与拓扑匹配的 `AllReduce`、`AllToAll` 等算法，以兑现物理拓扑带来的收益。
- **数据平面协议栈**：传输层可基于 `RoCEv2`，并关注 `UEC` 等面向 AI 场景的下一代以太规范；链路层则依赖 `PFC`、`ECN` 等无损以太能力，并结合 Torus 拓扑进行死锁与拥塞控制设计。

## 行业实践：谷歌 TPU

这一思路最有代表性的案例来自谷歌 TPU 网络。谷歌在以 OCS 为核心的智算网络中，采用了逻辑 3D Torus 作为高性能切片的主要形态。

在 TPUv4 架构中，基础构建单元是一个 `4 x 4 x 4` 的 TPU Cube，共 64 个 TPU 芯片。Cube 内部通过 PCB 背板与铜缆完成电信号直连，而位于 Cube 外表面的链路则通过光纤引出，接入 OCS 交换中心。谷歌使用多台 OCS 按 `X`、`Y`、`Z` 三个维度分组，在物理层完成不同维度的闭环连接，从而在逻辑上形成 3D Torus。

这种结构的关键不只是"能连起来"，而是可以按任务需求把一个大规模集群动态切成不同形状的 3D Torus 切片。更重要的是，当某个 TPU 芯片或链路失效时，控制器可以通知 OCS 在光层直接跳过故障节点，重新闭合 Torus 环，显著提升系统可用性。

在更大规模阶段，谷歌也进一步把这一思路从标准 3D Torus 推向更复杂的可重构形态。例如在新一代 TPU 系统中，通过 OCS 引入变步长的"虫洞式"跳跃连接，减少超大规模场景下 `AllToAll` 的跳数压力，这说明 OCS 的真正价值并不只是替代交换机，而是让"物理部署"与"逻辑通信拓扑"实现解耦。

## 架构优势

### 规模扩展

`OCS + 3D Torus` 通过网络扁平化和物理解耦，突破了传统电交换网络中端口数和层级持续增加的问题。新增算力单元接入 OCS 后，可以通过重构被动态纳入某个 Torus 切片，因此更适合沿着"更大超节点"方向平滑演进。

### 通信效率

OCS 结合 3D Torus 后，可以把逻辑上通信密集的节点对尽可能在物理层重构为更短路径，减少多跳电网络中的排队时延、拥塞和长尾问题。再叠加拓扑感知的集合通信算法，能进一步提升有效带宽利用率。

### 能效与成本

无源全光器件有助于降低核心层功耗，减少中间层的大量光电转换与包处理开销。由于 OCS 对速率和协议透明，计算节点和光模块代际演进时，核心基础设施的复用率也更高，有利于控制长期投资与升级成本。

### 可用性与弹性

当节点、链路或局部机柜出现故障时，OCS 可以在光层重构路径，快速绕过故障点，而不必完全依赖上层任务回收或人工重布线。这使其在超节点级别的容错和切片恢复上具有明显优势。

## 帕累托前沿定位

3D Torus + OCS 在帕累托空间中的位置可以概括为：**在拓扑弹性、故障隔离和核心层能效维度上占据前沿极值，在软件复杂度维度上付出最高代价**。

| 维度 | 帕累托位置 | 与其他方案的对比 |
|:-----|:----------|:---------------|
| 拓扑弹性 | ★★★★★ 全拓扑 OCS 重构 | Dragonfly+OCS 仅组间重构（★★★）；标准构型无重构（★） |
| 故障隔离 | ★★★★★ 光层绕障、切片恢复 | 其他方案依赖上层任务回收或人工重布线 |
| 功耗与 TCO | ★★★★★ 无源光层核心 | Dragonfly+OCS 仍有组内电交换；标准构型全电交换 |
| 规模上限 | ★★★★ 千–万卡（OCS 端口驱动） | Dragonfly+OCS ★★★★★；标准构型 ★★★ |
| 单跳延迟 | ★★★★ 直连光路无电交换中转 | 总线型 <1 μs（★★★★）；以太型 250–500 ns（★★） |
| 生态成熟度 | ★★★ Google TPU 验证，通用生态尚在建设 | 以太型 ★★★★★；Dragonfly+OCS ★★★ |
| 软件复杂度 | ★★ 拓扑管理+路由+集合通信全栈协同 | Dragonfly+OCS ★★★；以太型 ★★★★★ |

关键权衡：该方案将 OCS 从 Dragonfly+OCS 的"组间辅助"角色提升为"全拓扑核心"——OCS 不仅负责组间链路，而是负责构建整个 Torus 拓扑结构。这带来了所有探索构型中最强的拓扑弹性：可以按任务需求动态切出不同形状的 Torus 切片，也可以在光层直接绕过故障节点。但代价是软件栈必须从头设计——拓扑管理器、维度顺序路由、Torus 感知的集合通信算法都不能复用现有以太网组件。

与 Dragonfly + OCS 的不可支配关系：Torus + OCS 在拓扑弹性和故障隔离上严格优于 Dragonfly + OCS，但在软件复杂度和工程落地难度上严格劣于后者。两者在帕累托前沿上构成一对经典的不可支配关系——选择哪一个取决于组织对"弹性优先"还是"落地优先"的判断。

因此，该方案在帕累托前沿上的最佳适配区域是：**邻近通信模式为主（AllReduce 沿维度规约）、对故障隔离和切片可用性有严格要求、且组织具备全栈软件协同能力的大规模训练场景**。Google TPU 的实践证明了这一位置的可行性，但也说明了该方案对垂直整合能力的高要求。

## 展望

相较传统 `Clos` 架构，`3D Torus + OCS` 在拓扑弹性、故障隔离和 TCO 优化方面都展现出独特价值。随着 `CPO`、硅光集成与高速以太协议栈逐步成熟，这一路线有可能从"以太网端侧 + 光层重构"继续向"芯片出光 + 全光直连"演进，成为开放以太生态下构建全光超节点的一条重要探索路径。从帕累托演进的角度看，该方案代表了"最大化拓扑弹性"这条推进方向的当前极值——未来 dOCS 等更激进的方案将在此基础上进一步探索去交换化的前沿外推极限。
