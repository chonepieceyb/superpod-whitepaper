# 探索构型（大环路 + 分布式 dOCS）

大环路 + 分布式 `dOCS`（distributed Optical Circuit Switching）构型，是第四章里最激进的一条路线。它关注的已经不再是“如何在现有交换架构上继续优化”，而是更前沿的一组约束：**如果把光交换能力继续下沉到收发器级，去交换化是否有机会成为下一代高带宽域的默认组织方式。**

在这一路线中，`dOCS` 不再只是中心化 `OCS` 的缩小版，而是把“连接 + 动态交换”统一到收发器层，使单个光模块同时具备：

- 高速光互连传输能力
- 线路级/通道级动态交换与重构能力

这意味着系统不再默认依赖集中式交换机来定义拓扑，而是试图在端侧直接组织局部性边界。它把去交换化、极致能效和超低转发开销放在更前的位置，并接受由此带来的技术不确定性、控制面重构成本与工程验证压力。

## 锚定约束

当前主流 Scale-Up 构型，无论是总线型、以太型还是 `OCS` 辅助型，仍然默认“交换机”是组织高带宽域的中心器件。问题在于，当带宽继续向 `800G/1.6T` 乃至更高代际推进时，交换芯片、光电转换、板级布线、机柜供电与散热都会快速成为新的系统瓶颈。到这一步，系统要面对的已不只是"网络怎么更快"，而是**交换层本身是否正在成为局部性边界继续外推的约束。**

`dOCS` 的切入点正在这里：它不是在既有交换层上继续加速，而是试图减少乃至绕开集中式交换层，把高带宽域直接组织成由端侧器件共同完成的可重构光互连结构。若这一路线成立，系统可以把更多预算从中心交换设备转向端侧互连能力、拓扑控制和软件调度。

## 架构定义与组网方式

该构型可理解为一种"收发器中心（transceiver-centric）"的高带宽域组织方式。相较传统"端侧收发器 + 中心交换机"结构，它做了三个关键改变：

- **交换能力下沉**：交换不再主要发生在集中式交换芯片，而是部分下沉到分布式 `dOCS` 收发器中。
- **拓扑组织前移**：局部性边界不再由固定交换层预先决定，而是由端侧器件和控制器共同动态编排。
- **大环路互连**：系统更容易形成大平层、低中转、可重构的光路结构，把原本需要多跳包交换完成的路径，改写为更接近直达的电路式连接。

在实现形态上，可以将多颗具备 `dOCS` 能力的收发器部署在计算节点和机柜互连链路上，由控制器按任务流量和切片需求建立或调整大环路连接关系。与 `Dragonfly + OCS` 或 `Torus + OCS` 相比，这里不再由少数中心化 `OCS` 节点承担全局重构任务，而是由大量端侧器件共同参与拓扑形成。

这种结构的意义，不在于“少一层交换机”本身，而在于局部性边界的默认组织方式被改写了：系统开始把"交换"视为一种可以分布式完成的能力，而不是必须由少数核心设备独占的能力。

## 核心组件

### 分布式 dOCS 收发器

`dOCS` 收发器是该构型的核心。相较传统可插拔光模块，它不仅承担电光转换，还承担部分路径选择与光路切换能力。其关键价值在于把原本集中在交换层的能力部分前移到端侧。

### 硅光交换芯片

为了把交换能力做进收发器，`dOCS` 通常依赖更小型、更低功耗、更易集成的硅光交换芯片。与 `MEMS` 或液晶等中心化 `OCS` 技术相比，硅光路线更适合大规模端侧部署，也更接近未来 `CPO/NPO` 一体化演进方向。

### 高密度光互连与封装

一旦交换能力下沉，系统对光纤管理、端口密度、光电协同封装与维护体系的要求会显著上升。因此，这条路线能否成立，不只取决于交换原理，也取决于封装、布线、模块可维护性和热设计是否同步成熟。

## 软件栈与控制平面

`dOCS` 构型的真正难点并不只在器件，而在控制面。因为当交换能力被分布式下沉后，系统必须重新定义"谁来维护拓扑一致性、谁来决定路径、谁来感知故障、谁来触发重构"。典型软件职责包括：

- **全局拓扑控制器**：根据任务规模、流量画像和链路健康状态，决定大环路的形成与重构策略。
- **端侧交换管理器**：把全局拓扑意图下发到分布式 `dOCS` 收发器，协调大量端侧器件的局部动作。
- **拓扑感知调度器**：把模型切片、通信模式与当前光路结构绑定，避免“物理可重构但任务不感知”的空转。
- **容错与回退机制**：在端侧交换失配、模块异常或局部链路退化时，快速回退到可用拓扑，避免分布式失配放大为全局故障。

与 `Torus + OCS` 相比，`dOCS` 的控制面并不是更简单，而是把复杂度从中心设备迁移到了大量端侧器件和控制器协同上。它代表的是一种更激进的复杂度分布方式。

## 行业实践与前沿来源

这一路线目前仍主要来自前沿研究而非成熟产品。`SIGCOMM 2025` 的《InfiniteHBD: Building Datacenter-Scale High-Bandwidth Domain for LLM with Optical Circuit Switching Transceivers》系统性提出了 `transceiver-centric HBD architecture` 的概念，核心判断正是：与其继续依赖"收发器点到点 + 中心交换机动态调度"的架构，不如直接让收发器成为高带宽域动态组织的一部分。

这使 `dOCS` 与本章其他方案形成了明确分工：它不是为了替代今天的标准构型，而是为了测试当光交换能力进一步端侧化之后，下一代高带宽域是否可以摆脱对大型交换芯片的依赖。

## 架构优势

### 去交换化潜力

该构型最大的想象空间，在于有机会减少集中式交换层级，降低中转、排队与光电转换开销，把更多预算留给端侧带宽与系统功耗优化。

### 能效潜力

如果端侧硅光交换与高密度封装成熟，系统有望显著降低核心交换层功耗，并减少因多级转发带来的附加开销。这使其在长期 `TCO` 和能效维度上具备很强的前沿外推价值。

### 边界重构自由度

由于交换能力更靠近端侧，系统在理论上可以更细粒度地按任务、切片和链路健康状态去组织局部性边界，而不必完全受限于固定交换层的层级结构。

## 帕累托前沿定位

从取舍结构看，大环路 + `dOCS` 更靠近这样一侧：**去交换化潜力、能效前景和转发开销压缩更强，但成熟度、控制面复杂度和工程确定性代价最高**。

| 维度 | 帕累托位置 | 与其他方案的对比 |
|:-----|:----------|:---------------|
| 去交换化潜力 | ★★★★★ 交换能力下沉到端侧 | 其余方案仍默认依赖中心交换层 |
| 功耗与 TCO 潜力 | ★★★★★ 长期前景最强 | 受制于器件成熟度，短期未必兑现 |
| 转发开销 | ★★★★★ 理论上最小 | 省去多级包交换和部分中转 |
| 拓扑弹性 | ★★★★★ 端侧可重构 | 强于标准构型，接近或超越集中式 OCS |
| 生态成熟度 | ★ 研究验证阶段 | 明显低于所有现有工程路线 |
| 软件复杂度 | ★ 需要重建控制面 | 复杂度分布最激进 |
| 工程确定性 | ★ 器件、封装、运维均待验证 | 不适合作为近期标准构型 |

关键权衡在于：`dOCS` 并不是把复杂度消灭了，而是把复杂度从中心交换机和固定拓扑，迁移到了端侧器件、封装与控制器协同之中。它因此不是当下最稳妥的部署方案，而是一种面向下一代边界外推的前沿押注。

因此，该方案在帕累托前沿上的最佳适配区域是：**以中长期技术验证为目标、具备光器件/封装/控制面协同能力、希望提前布局去交换化高带宽域的组织与研究平台**。对于绝大多数近期落地项目，它更适合作为并行验证方向，而不是主交付路线。

## 展望

如果未来 `CPO/NPO`、硅光交换、端侧控制与拓扑感知调度共同成熟，`dOCS` 可能把高带宽域的默认组织方式从"中心交换主导"改写为"端侧器件协同主导"。从帕累托演进的角度看，它并不只是又一个探索网络，而是在测试一个更根本的假设：**下一代超节点是否可以把交换能力从网络中心重新分发回系统边缘。**
