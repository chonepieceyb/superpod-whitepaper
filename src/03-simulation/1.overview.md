# 1. 建模仿真概述

随着人工智能大模型（LLMs）向万亿参数与万卡集群演进，AI 基础设施的系统复杂度已超越了常规规划方法的处理极限。建模仿真不再局限于学术研究范畴，而是转变为贯穿 AI 基础设施**规划、建设、维护、优化全生命周期**的核心数字化底座。

## 1.1 全生命周期的数字化底座

大模型系统的建设是一个持续迭代的动态演进过程。建模仿真体系通过构建物理系统的“数字孪生”（Digital Twin），在全生命周期的各个阶段提供关键支撑：

* **规划期（Planning）**：支持技术选型与架构设计决策。在采购实施前，对不同硬件组合（如网络拓扑架构、算力资源配比）的理论收益进行预先评估。
* **建设期（Construction）**：在物理集群部署交付之前，提前进行分布式并行策略的逻辑验证与调度参数的预调优，从而缩短系统交付周期。
* **维护期（Maintenance）**：作为故障演练的虚拟沙盒，模拟网络链路中断、计算节点失效等异常工况，评估系统的鲁棒性与故障恢复机制。
* **优化期（Optimization）**：在业务运行阶段，持续在参数空间中搜索更优的并行策略与调度配置，挖掘存量硬件资源的极致效能。

## 1.2 核心场景：从数据推断到假设分析

从抽象维度审视，本白皮书所定义的建模仿真体系主要解决以下三大核心场景问题：

1.  **性能推演（Performance Projection）**：
    * **定义**：利用可获取的原子级指标，推演难以直接测量的系统级性能表现。
    * **场景**：基于单卡微基准测试数据（低获取成本）与负载逻辑图，推演万卡集群在特定并行策略下的端到端训练吞吐量（高获取成本、高测量难度）。

2.  **假设分析（What-if Analysis）**：
    * **定义**：针对反事实假设进行量化评估，在零物理成本的前提下探索系统的性能边界与可行性空间。
    * **场景**：评估关键资源变量变化对系统效率的影响（例如：量化分析网络带宽缩减对训练效率的敏感度；评估 Batch Size 增大对显存容量约束的潜在冲击），从而预演物理世界中高风险或高成本的操作方案。

3.  **高效评估与优化（Efficient Evaluation & Optimization）**：
    * **定义**：利用仿真器的高速推演能力，快速遍历海量输入参数组合，定位全局最优配置解。
    * **场景**：在包含 TP/PP/DP 切分方式、通信计算重叠策略、缓存调度参数的高维搜索空间中，通过高并发仿真快速筛选出满足服务等级协议（SLA）约束的最佳系统配置。

## 1.3 建模仿真方法论

为支撑上述场景，本白皮书确立了**“客观映射、维度递进、资产驱动”**的方法论体系。

### 1.3.1 客观映射体系（Objective Mapping System）
系统仿真本质上是一个确定性的演绎引擎。它构建了一套从**输入空间**（负载需求、硬件能力、系统配置）到**输出空间**（系统行为）的客观映射关系：

$$\text{Simulation}(Workload, Resource, Strategy) \rightarrow \text{System Behavior}$$

该引擎不预设主观偏好，而是依据严谨的因果逻辑，如实反映数据在系统中的流动、计算与交互过程。输出的**“系统行为”**不仅包含量化的**性能指标（Metrics）**，还涵盖系统的**逻辑状态（State）**与全过程**执行轨迹（Trace）**。

### 1.3.2 仿真维度的递进：功能、性能与行为
本体系遵循严格的验证次序，从三个维度对系统进行全方位评估：

1.  **功能仿真（Functionality）**：**功能正确性验证**。确保计算图的依赖关系准确，数据切分逻辑与张量形状匹配，并验证系统中不存在死锁或资源竞争导致的逻辑冲突。
2.  **性能仿真（Performance）**：**性能效率评估**。量化系统的吞吐量（Throughput）、端到端延迟（Latency）、硬件资源利用率（Utilization）以及流水线执行效率。
3.  **行为仿真（Behavior）**：**系统鲁棒性分析**。模拟故障注入（Fault Injection）、网络拥塞震荡、长尾延迟等非理想状态下的系统响应机制与稳定性表现。

### 1.3.3 层次化建模架构：需求-映射-供给

为解决大规模分布式系统的复杂耦合挑战，本白皮书采用**“正交解耦”**的建模方法论。我们将仿真对象划分为三个逻辑层次，形成从逻辑需求到物理供给的闭环映射体系：

1.  **负载层（Workload Layer - 需求侧）**
    * **定义**：描述应用在逻辑层面的**算力需求**。
    * **核心职能**：定义逻辑计算图（Computation Graph）与张量数据流语义。该层严格遵循**硬件无关（Hardware-agnostic）**与**策略无关（Strategy-agnostic）**原则，仅描述计算任务的数学属性与依赖关系，剥离所有关于运行位置、并行切分方式或执行时间的物理属性，确保负载描述的纯粹性与可复用性。

2.  **系统层（System Layer - 映射引擎）**
    * **定义**：描述需求与供给之间的**映射关系**。
    * **核心职能**：作为仿真器的核心推演引擎，负责将抽象的负载“实例化”到具体的物理资源上。其核心职责包括**并行策略映射（Mapping）**（如 TP/PP/DP 的切分决策）与**流水线编排（Scheduling）**。该层决定了逻辑算子在物理设备上的时空分布，是连接软件需求与硬件供给的关键桥梁。

3.  **资源层（Resource Layer - 供给侧）**
    * **定义**：描述物理硬件的**性能边界**。
    * **核心职能**：对计算、存储、网络子系统进行抽象建模，构建**性能资产库（Performance Asset Library）**。该层不处理复杂的调度逻辑，而是提供高保真的原子级性能数据（如算子执行延迟、链路有效带宽），回答“特定操作在特定硬件上的物理开销”这一基础定量问题。
