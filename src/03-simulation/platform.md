# 资源模型：把供给侧边界从规格表里解放出来

如果说负载模型回答的是“系统会承受什么压力”，那么资源模型要回答的就是：**当这些压力真正落到基础设施上时，供给侧究竟能提供怎样的能力边界，又会从哪里开始付出代价。** 一个常见写法，是先罗列平台，再罗列参数，最后补几张测试曲线；但对超节点产业真正重要的，从来不是“平台有什么”，而是“平台在什么条件下还能兑现这些能力，以及从哪里开始失真、拥塞、抖动和退化”。

这也是资源模型不能停留在静态规格参数上的原因。峰值算力、链路带宽、`HBM` 容量和单跳时延当然重要，但如果分析止步于此，就无法解释为什么很多纸面更强的路线，在真实系统里反而离可达前沿更远。真正会改写边界的，不只是芯片和链路上限，还包括运行时开销、调度组织能力、恢复路径、跨平面互扰、碎片化以及设施侧约束。换句话说，供给侧不是一组孤立硬件对象，而是一整套“**系统在长期运行中能够稳定供给什么**”的能力结构。

## 资源模型真正描述的，是能力边界和代价曲面

从整章主线看，资源模型至少要同时回答两件事。第一，上界在哪里，也就是计算、访存、互联、运行时和恢复能力的理论边界；第二，这些上界会从哪里开始被侵蚀，也就是拥塞、分页、排队、抖动、重传、重平衡和恢复代价如何逐步吃掉这些能力。只有前者，会把分析写成理想硬件图景；只有后者，又会失去可比较的统一基线。资源模型的价值，正在于把二者组织成同一张供给侧地图。

因此，真正重要的资源对象，并不是按“硬件类别”被挑出来的，而是按“是否会改变边界形状”被挑出来的。下表给出五类核心资源对象及其边界塑形机制：

| 资源对象 | 理论上界参数 | 代价曲面（性能开始退化的机制） | 对前沿的塑形作用 |
|:---------|:-----------|:---------------------------|:---------------|
| 计算资源 | 峰值 FLOPS（FP16/FP8/INT8）、SM/CU 数量、功耗墙 | 精度降级损失、功耗限流、热降频、多租户争用 | 决定算子执行效率与单位能耗边界 |
| 存储资源 | HBM 容量与带宽、DDR 容量、NVMe 交换带宽 | 分页碎片化、缺页代价、HBM-DDR 交换长尾、驱逐抖动 | 决定显存可行域和 KV Cache 管理效率 |
| 互联资源 | 链路带宽（NVLink/PCIe/Ethernet）、端口数、拓扑 | 拥塞窗口收缩、多路径不均衡、重传代价、跨平面干扰 | 决定通信吞吐和同步放大程度 |
| Runtime 资源 | 启动延迟、调度周期、内存分配开销 | kernel launch 累积、调度排队、allocator 碎片化 | 侵入关键路径时直接降低 MFU 和增大气泡率 |
| RAS 资源 | 故障率（FIT）、检测延迟、隔离粒度、MTTR | 恢复路径引入的停机时间、checkpoint 写入开销、回稳抖动 | 决定长期 Goodput 与理论峰值之间的折扣系数 |

把 `Runtime` 和 `RAS` 视作资源，而不是附注，正是超节点建模与传统设备评测的重要分界之一。

在超大规模系统里，还有一批传统规格表不会主动交代、但资源建模必须显式接住的系统级变量：

| 系统级变量 | 影响路径 | 如果不纳入会怎样 |
|:----------|:--------|:---------------|
| 多平面网络组织（参数面/数据面/业务面） | 决定哪些流量会侵入主路径 | 低估通信长尾和跨平面干扰 |
| 统一调度（粒度、周期、重平衡延迟） | 决定理论资源何时变为可执行资源 | 高估资源利用率，低估调度开销 |
| 长期运行状态（故障率、碎片率、恢复时间） | 决定长期 Goodput 距离理论值多远 | 把稳态快照误判为长期表现 |
| 设施侧约束（机柜功率、液冷能力、PUE、建设周期） | 决定设计点是否可部署 | 把纸面最优当成工程可达 |

若不把这些变量一起纳入，所谓“系统可达边界”就很容易被误写成“局部硬件极限”[^ultra-report-platform-scope]。

## 不是所有资源事实都值得用同样成本获取

资源模型因此不能只是分散测试结果的堆积，而必须是一套可调用、可扫描、可校准的供给侧表达。系统模型真正需要查询的，不是“你做过多少实验”，而是：在给定 `shape`、精度、`batch` 和并发占用下，算子执行时间和功耗大致是多少；在给定层次与访问模式下，带宽、时延、缺页代价和交换代价如何变化；在给定拓扑、消息大小和并发度下，通信耗时与拥塞开销大概落在哪里；当调度与恢复被触发时，固定开销和动态开销会怎样侵入执行路径。只有当这些资源事实被沉淀为可调用对象时，资源模型才真正进入整章主链。

这也意味着，资源模型必须在保真度上分层。超节点级分析面对的是数千点甚至更大的候选空间，高精度仿真和端到端测量不可能直接承担全空间描边。下表给出三级保真度及其在工作流中的位置：

| 保真度层级 | 数据来源 | 单点成本 | 适用阶段 | 产出 |
|:----------|:--------|:---------|:---------|:----|
| L1 解析层 | 公开规格、协议上界、Roofline 公式、TCO 参数 | 极低（毫秒级） | 全空间快速扫描，排除被支配区域 | 前沿轮廓、被支配域剔除 |
| L2 画像层 | 微基准、子系统压测、collective benchmark、trace 拟合 | 中等（分钟级） | 甜点区识别、主要方案对比 | 局部前沿收紧、主要排序判断 |
| L3 行为层 | 细粒度事件仿真、故障注入、控制面日志、拥塞演练 | 高（小时级） | 候选前沿点、跳变点、异常点校准 | 偏差解释、置信区间标定 |

这种分层不是为了妥协，而是为了尊重问题规模。若把所有资源对象都用同等高保真度去描述，描边本身会在成本上失去可行性；若完全停留在最低保真度，边界虽然能画出来，却难以支撑排序和归因。资源模型最合适的姿态不是“越细越好”，而是“先足够便宜地覆盖大范围，再足够有针对性地收紧关键区域”[^ultra-report-platform-runtime]。

## 性能资产库：资源模型的具体承载形态

这也是为什么性能资产库不是附属平台功能，而是资源模型的具体承载形态。公开规格、论文、实验室微基准、线上遥测、故障演练和业务 `trace` 之所以要被统一纳入资产库，不是为了数据治理本身，而是为了让资源画像具备版本化、可追溯和可查询能力。没有资产库，资源模型只能停留在零散表格；有了资产库，系统模型才知道某条带宽曲线来自哪一代硬件、哪个软件版本、哪种测试条件，以及它在什么适用边界内可以被使用。

性能资产库的最小可行结构应至少包含以下字段：

| 字段 | 说明 | 示例 |
|:----|:----|:----|
| 资源类型 | 计算/存储/互联/Runtime/RAS | `compute.gemm` |
| 硬件平台 | 芯片型号 + 系统配置 | `H100 SXM5 × 8, NVLink 4.0` |
| 软件版本 | 驱动、CUDA、框架、通信库版本 | `CUDA 12.4 + NCCL 2.21` |
| 测试条件 | shape, 精度, 并发度, 消息大小等 | `M=4096, K=4096, N=8192, FP16` |
| 保真度层级 | L1/L2/L3 | `L2` |
| 数值 | 测量或估算结果 | `312 TFLOPS, 1.8ms` |
| 适用边界 | 该数据在什么条件下可被外推 | `batch ≤ 256, 单节点内` |
| 来源与时间 | 论文/实测/线上遥测 + 采集日期 | `内部实测, 2025-11` |

至此，资源模型的行业角色也就清楚了。它不是硬件百科，也不是评测体系的替代物，更不是未来路线图的前置摘要。它的任务只有一个：把供给侧能力边界和代价曲面参数化，让系统模型能够在可承受的成本下**先描边，再校准**。也因此，资源模型必须保持克制，只回答“系统有哪些能力与代价可供建模”，而不越界去做方案排序或未来判断；那些判断应当留给系统模型、校准验证以及参考设计比较与未来演进分析完成。

同样，案例之所以能够说明问题，也不是因为它简单比较了两类 GPU，而是因为它把这里定义的供给侧边界真正带入了阶段瓶颈、带宽约束和成本曲面的统一判断中。案例验证的不是硬件参数本身，而是这套资源建模视角是否足以支撑方案比较。
